Authorization Boundary Evidence Packet
What jabearri verifies
jabearri answers one question: does a protected resource return the same response to a different authenticated user?

It records real authenticated traffic from user A, replays each request using user B's token, and compares responses using content hashes. If the endpoint is expected to be owner-scoped and the replayed response matches, jabearri flags a confirmed authorization-boundary failure. No raw tokens or response bodies are stored. Response values are summarized by field path, classification signal, size, and content hash. Endpoint paths and resource identifiers are retained only where needed for reproduction.
How a run works
01 Record authenticated requests from user A against target endpoints in a sandbox environment.
02 Replay each request using user B's token. No modifications to the request — only the bearer token changes.
03 Hash both responses using a stable semantic algorithm. Compare. If hashes match on an owner-scoped endpoint: boundary failed.
04 Produce a reproducible evidence record with finding ID, exposure summary, and a reproduction command for independent verification.

What a pilot looks like

Scope

The goal is not a broad pentest. The goal is to verify whether specific owner-scoped API objects — accounts, wallets, payments, loans, patient records, provider records, tenant-scoped resources — can cross user or account boundaries under authenticated replay.
  • 3–5 read-only endpoints in your sandbox or staging environment
  • Two test accounts you create and control — no access to production data
  • Explicit written authorization before any run begins
  • Evidence report delivered in the format shown above
  • Reproduction command for every confirmed finding
  • One follow-up call to walk through findings and answer technical questions
